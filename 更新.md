> 基于文章《一文读懂 Agent Tools，拒绝复杂化、碎片化、黑盒化》的核心理念，对 Shoplive 项目进行 Agent Tools 层面的系统性优化。
>
> 日期：2026-02-27

## 一、优化背景

原文核心观点：**合格的 Agent Tool 应该是一个"可理解、安全且具备容错能力"的交互接口。** 具体体现在 5 个生命周期阶段的关键要素：

1. 类型安全与自动化
2. LLM 友好的接口设计
3. 自我修复能力（结构化错误 + 修复建议）
4. Human-in-the-loop 安全防护
5. 性能优化与上下文管理

以及 3 个企业级痛点的解决：**碎片化、复杂化、黑盒化**。

## 三、详细变更说明

### 3.1 Pydantic Schema 类型验证层

**文件**: `backend/schemas.py` (新建)

**对应原则**:
> "充分利用 Python 类型系统和 Pydantic，自动处理 schema 生成和数据验证，防止模型瞎猜。"

**变更内容**:

为所有 API 端点创建了 Pydantic BaseModel 类：

- `ProductInsightRequest` — 商品链接解析
- `ImageInsightRequest` — 商品图片分析
- `AgentChatRequest` — LLM 对话
- `VideoWorkflowRequest` — 视频工作流
- `VeoStartRequest` — Veo 视频生成
- `VeoChainRequest` — 链式视频生成
- `VeoStatusRequest` — 视频状态查询
- `VeoExtendRequest` — 视频延展
- `VideoEditExportRequest` — 视频编辑导出
- `ShopliveImageGenerateRequest` — 商品图片生成

关键设计点：

```python
# 1. Literal 限制枚举值，减少模型出错概率
language: Literal["zh", "en"] = Field(default="zh", ...)
veo_mode: Literal["text", "image", "reference"] = Field(default="text", ...)
duration_seconds: Optional[Literal[4, 6, 8]] = Field(default=None, ...)

# 2. 自然语言 Field descriptions，让 LLM 理解如何传参
product_url: str = Field(
    description="Full product page URL starting with http:// or https://. "
                "Example: 'https://www.amazon.com/dp/B0XXXXXX'"
)

# 3. 清晰的默认值，减轻模型负担
sample_count: int = Field(default=1, ge=1, le=4, description="...")
```

**依赖变更**: `requirements.txt` 新增 `pydantic>=2.0`

### 3.2 结构化错误响应 + 修复建议

**文件**: `backend/common/helpers.py`, `backend/api/agent_api.py`, `backend/api/veo_api.py`

**对应原则**:
> "工具不应在遇到错误时直接抛出异常导致流程终止，而应引导 Agent 调整策略。结构化错误返回包含 error 信息和 recovery_suggestion（修复建议）。"

**变更内容**:

1. **增强 `json_error()` 函数**，新增 `recovery_suggestion` 和 `error_code` 参数：

```python
# 之前
return json_error("prompt 不能为空")

# 之后
return json_error(
    "prompt 不能为空",
    recovery_suggestion="Use /api/shoplive/video/workflow with "
                        "action='build_export_prompt' to auto-generate a prompt.",
    error_code="MISSING_PROMPT",
)
```

2. **预定义 `RECOVERY_HINTS` 常量**，覆盖所有常见错误场景的修复建议。

3. **为所有 API 端点的错误返回添加修复建议**：

| 错误场景 | error_code | recovery_suggestion |
|---------|-----------|-------------------|
| 缺少商品链接 | MISSING_PRODUCT_URL | 提供 http/https 链接，列出支持平台 |
| 缺少图片输入 | MISSING_IMAGE | 提供 image_items/image_base64/image_url 之一 |
| 缺少 API Key | MISSING_API_KEY | 通过 payload 或 LITELLM_API_KEY 环境变量提供 |
| 缺少 Prompt | MISSING_PROMPT | 使用 video/workflow 自动生成 prompt |
| 抓取失败 | SCRAPE_FAILED | 检查 URL/代理，或改用 image-insight |
| Veo 提交失败 | VEO_SUBMIT_FAILED | 检查网络/凭证/模型名称 |

### 3.3 LLM 友好的 Tool Registry

**文件**: `backend/tool_registry.py` (新建), `backend/web_app.py`

**对应原则**:
> "LLM 无法像传统程序那样通过技术文档理解接口，它依赖于自然语言描述来决定如何使用工具。" + "工具名称用动词-名词结构，描述既要简洁又要完整说明用途。"

**变更内容**:

1. **创建 `TOOL_REGISTRY`**，包含 10 个工具的标准化定义：

```python
{
    "name": "parse_product_url",           # 动词-名词命名
    "display_name": "Parse Product URL",
    "description": "自然语言描述，说明用途和场景",
    "endpoint": "POST /api/agent/shop-product-insight",
    "tags": ["product", "scraping", "analysis"],
    "skill": "product_analysis",           # 所属技能
    "parameters": { ... },                 # 参数 schema
    "output_summary": { ... },             # 输出摘要（非全文）
    "examples": [ ... ],                   # 示例调用
    "common_errors": { ... },              # 常见错误及处理
    "next_tools": [ ... ],                 # 后续工具建议
}
```

2. **支持智能召回机制**（对应文章"智能召回"概念）：

- `get_tools_by_skill(skill_name)` — 按技能筛选
- `get_tools_by_tags(tags)` — 按标签筛选
- `build_tool_manifest()` — 完整清单

3. **新增 API 端点** `GET /api/tools/manifest`：

```
GET /api/tools/manifest                     → 全部工具
GET /api/tools/manifest?skill=video_generation → 视频相关工具
GET /api/tools/manifest?tags=product,analysis  → 商品分析工具
```

### 3.4 Skills 技能编排层

**文件**: `backend/skills.py` (新建), `backend/web_app.py`

**对应原则**:
> "我们将工具升级为技能（Skills）。" + "对于复杂流程，用 Agent Skill 写一份操作说明书，引导模型按步骤调用多个工具完成任务。"

**变更内容**:

1. **定义 4 个核心 Skill**：

| Skill ID | 名称 | 包含工具 | 用途 |
|----------|------|---------|------|
| `product_to_video` | Product URL to Video | 4 必需 + 3 可选 | 从商品链接到成品视频的端到端流程 |
| `image_to_video` | Product Image to Video | 4 必需 | 从商品图片到视频的流程 |
| `batch_product_images` | Batch Product Image Generation | 1 必需 | 批量生成商品图片 |
| `video_post_production` | Video Post-Production | 1 必需 | 视频后期编辑 |

2. **每个 Skill 包含完整的 `execution_guide`**（操作说明书），以 Markdown 格式详细说明：
   - 前置条件
   - 分步操作指引（含完整的 API 调用示例）
   - 错误处理策略

3. **渐进式披露** API 设计：

```
GET /api/skills           → 技能摘要列表（轻量）
GET /api/skills/<id>      → 完整技能定义 + execution_guide（按需加载）
```

## 九、第三轮优化（2026-02-28）——Pydantic 请求验证中间件

> 将第一轮建好但未接入的 Pydantic schema，通过统一装饰器在**系统边界**强制校验，完成类型安全的最后一公里。

### 9.1 新建 `backend/validation.py`

**对应原则**：
> "类型安全——在任何业务逻辑执行前，于系统边界校验所有输入。"

**核心实现**：`validate_request(schema_cls)` 装饰器

```python
@app.post("/api/agent/shop-product-insight")
@validate_request(ProductInsightRequest)   # ← 在 handler 执行前校验
def api_agent_shop_product_insight():
    req = g.req   # 已验证的 Pydantic 模型实例
    product_url = req.product_url   # 类型安全，URL 格式已校验
    ...
```

**校验失败时**返回结构化 400，LLM 可直接自修正：

```json
{
  "ok": false,
  "error": "Request validation failed: 'product_url' — Value error, product_url must be a valid http/https URL",
  "error_code": "VALIDATION_ERROR",
  "recovery_suggestion": "Fix 'product_url': ... See GET /api/openapi.json → components/schemas/ProductInsightRequest.",
  "validation_errors": [
    {"field": "product_url", "message": "...", "type": "value_error"}
  ]
}
```

**关键设计**：
- `recovery_suggestion` 指向 `/api/openapi.json`，将 Pydantic schema、OpenAPI 规范和运行时错误三者打通
- `functools.wraps` 保留原函数名，不破坏 Flask 路由注册
